# UIレビュー ベストプラクティス

## レビューの基本方針

UIレビューでは以下の4軸を基本観点とする:
- 視認性（visibility）: テキストの読みやすさ、コントラスト、フォントサイズ
- 情報構造（information_structure）: 見出し階層、グルーピング、情報の優先度
- 操作導線（navigation）: CTA の配置、ボタンの視認性、遷移の分かりやすさ
- 入力負荷（input_burden）: フォームの項目数、入力補助、エラー表示

## ツールの使い分け

### analyze_ui_image を使う場面
- ローカル開発環境のスクリーンショットをレビューしたい
- デザインカンプ（Figma等のエクスポート画像）をレビューしたい
- 特定の状態（エラー表示、ローディング等）のスクリーンショットをレビューしたい

### analyze_ui_url を使う場面
- 公開されているWebページをレビューしたい
- DOM構造やアクセシビリティ情報も含めた詳細なレビューが必要
- axe-core による自動検査結果が欲しい

### generate_ui_fix_prompt を使う場面
- レビュー結果を実装AIに渡して修正を依頼したい
- 技術スタックに合わせた具体的な修正指示が必要
- 複数のレビュー結果を統合して一括修正したい

## 効果的なレビューのコツ

### purpose パラメータを活用する
画面の目的を明示すると、その目的に沿ったレビュー観点が強化される。

良い例:
- "会員登録の完了率改善"
- "商品検索の離脱率低減"
- "ダッシュボードの初回理解度向上"

悪い例:
- "UIを良くする"（曖昧すぎる）
- 未指定（汎用的なレビューになる）

### デバイス指定を忘れない
モバイルとデスクトップでは評価基準が異なる。
対象デバイスを正しく指定することで、適切なビューポートとレビュー観点が適用される。

### generate_ui_fix_prompt の techStack を正確に
技術スタックを正確に指定すると、そのスタックに適した修正指示が生成される。

例: "Next.js / React / Tailwind CSS / shadcn/ui"
例: "Laravel / Blade / Tailwind CSS"
例: "Vue 3 / Nuxt 3 / UnoCSS"

## 注意事項

- このツールはUIの問題を「断定」しない。レビュー材料を整理してホスト側LLMに渡す役割
- axe-core の結果は参考情報であり、すべてのアクセシビリティ問題を検出できるわけではない
- 生成された改善指示プロンプトは、実装前に人間がレビューすることを推奨
